## 1. なぜGitHubを導入するのか

### 従来の課題

```
【これまでの業務】

📁 Google Drive / ローカルPC
    │
    ├─ 議事録_20250115.docx
    ├─ 議事録_20250122.docx
    ├─ 提案資料_v2_最終_本当の最終.pptx
    └─ ...

        ↓ 問題点

❌ AIに「過去の経緯を踏まえて」と頼めない
❌ 誰がいつ何を変えたか追えない
❌ 案件の全体像を把握するのに時間がかかる
❌ 情報が散らばって探せない
```

### GitHubで解決できること

```
【GitHubを導入すると】

📁 GitHub + Cursor（AI）
    │
    ├─ 全ファイルをAIが読める
    ├─ 変更履歴が完全に残る
    ├─ チーム全員がリアルタイムで共有
    └─ 構造化されたデータで検索性向上

        ↓ 結果

✅ 「過去の議事録を踏まえて提案作成して」が可能
✅ 「前回からの進捗をまとめて」が一瞬
✅ 案件の引き継ぎが圧倒的に楽になる
✅ AIが案件のコンテキストを理解して回答
```

## 2. GitHubで実現できる未来

### Before → After

| 業務 | Before（従来） | After（GitHub + AI） |
|------|----------------|----------------------|
| 議事録作成 | 過去のメモを手動で探す | AIが過去の議事録を参照して下書き作成 |
| 提案書作成 | ゼロから考える | 「前回の効果検証を踏まえて次の提案を」 |
| 進捗報告 | 各所から情報を集める | 「今月の進捗をまとめて」で一発 |
| 引き継ぎ | 膨大な説明が必要 | AIが案件概要を説明してくれる |
| ナレッジ共有 | 属人化 | 全員がアクセス可能 |

### 具体的なAI活用シーン

#### シーン1：MTG前の準備

```
あなた：「クライアントAの過去3回の議事録を読んで、
        主な懸念点と決定事項をまとめて」

AI：過去の議事録を分析...

【主な懸念点】
- CVRの改善が見られない（1/15, 1/22で言及）
- 競合サイトの順位上昇（1/22で報告）

【決定事項】
- KW選定の見直し（担当：山田、期限：1/31）
- LP改善案の提出（担当：鈴木、期限：2/5）
```

#### シーン2：提案書の作成

```
あなた：「先月の効果検証レポートと施策進捗を踏まえて、
        来月の提案内容を考えて」

AI：ファイルを分析...

【提案の方向性】
1. 効果が出ているKW群の横展開
   - 「〇〇 比較」系KWでCVR 3.2%達成
   - 類似KW「△△ 比較」「□□ おすすめ」へ展開

2. 改善が必要な施策の見直し
   - 内部リンク施策の効果が限定的
   - 代替案：ピラーページ構造への変更を提案
```

#### シーン3：新メンバーへの引き継ぎ

```
新メンバー：「クライアントAの案件について教えて」

AI：案件フォルダを分析...

【案件概要】
- クライアント：株式会社〇〇
- 契約開始：2024年10月
- 月額：50万円
- 主なKPI：オーガニック流入数、CV数

【現在の状況】
- 記事30本公開済み、月間10本ペースで追加中
- 主要KW「〇〇」で3位→1位に改善
- 課題：CVRの改善が次のフェーズ

【直近のアクション】
- 1/29 効果検証MTG実施
- 2/5 LP改善案提出予定
```

## 4. GitHubとは

GitHubは「ファイルの共有・バージョン管理サービス」です。

### 特徴

| 特徴 | 説明 |
|------|------|
| チームで共有 | 全員が同じファイルにアクセスできる |
| 履歴が残る | 誰がいつ何を変更したか記録される |
| 元に戻せる | 間違えても過去のバージョンに戻せる |
| AIと連携 | Cursor等のAIツールが全ファイルを読める |
| 無料で使える | 基本機能は無料 |

### Google Driveとの違い

| | Google Drive | GitHub |
|---|---|---|
| 同期 | 自動・リアルタイム | 手動（Push/Pull） |
| 履歴 | 簡易的 | 詳細に記録 |
| AI連携 | ❌ 難しい | ✅ 最適 |
| 向いている用途 | 一般ファイル共有 | ドキュメント・ナレッジ管理 |

### なぜGoogle DriveではダメなのかAI活用において、Google Driveには構造的な限界があります。

```
【Google Drive + AI】

📄 Google Docs形式（.gdoc）
    ↓
AIツール（Cursor等）が直接読み込めない
    ↓
毎回コピペが必要
    ↓
「過去の文脈を踏まえて」ができない
```

```
【GitHub + AI】

📄 マークダウン形式（.md）
    ↓
AIが全ファイルを自動で読み込める
    ↓
「過去の議事録を踏まえて提案して」が可能
    ↓
案件の全体像を理解した上で回答
```

## 7. 基本操作（ブラウザ編集）

PCにソフトをインストールせず、ブラウザだけで編集できます。

### ファイルを見る

1. GitHubでリポジトリを開く
2. 見たいファイルをクリック

### ファイルを編集する

1. ファイルを開く
2. **鉛筆アイコン（Edit）** をクリック
3. 内容を編集
4. 下の「Commit changes」をクリック
5. メッセージを入力（例：「チェック項目追加」）
6. 「Commit changes」で保存

```
┌─────────────────────────────────────────────┐
│ 📄 チェックリスト.md                         │
│                                     [✏️]    │
│                                      ↑      │
│                               鉛筆アイコン   │
└─────────────────────────────────────────────┘
```

## 8. 基本操作（VSCode / Cursor編集）

AIを活用した編集にはCursor（またはVSCode）が必要です。

### 初回設定：クローン（ダウンロード）

1. GitHubでリポジトリを開く
2. 「Code」ボタン → URLをコピー
3. Cursor（またはVSCode）を開く
4. `Cmd + Shift + P`（Mac）または `Ctrl + Shift + P`（Windows）
5. 「Git: Clone」と入力して選択
6. URLを貼り付け
7. 保存先フォルダを選択
8. 完了

### 日常の作業フロー

```
┌─────────────────────────────────────────────┐
│           毎日の作業フロー                    │
├─────────────────────────────────────────────┤
│                                             │
│  【作業開始】                                │
│     Pull（最新を取得）                       │
│       ↓                                     │
│  【作業中】                                  │
│     編集 → 保存 → ステージ → コミット         │
│       ↓                                     │
│  【作業終了】                                │
│     Push（GitHubに反映）                     │
│                                             │
└─────────────────────────────────────────────┘
```

### Pull（最新を取得）

他のメンバーの変更を自分のPCに反映します。

**操作方法：**
- 左下の 🔄 アイコンをクリック
- または Source Control →「...」→「Pull」

**タイミング：** 作業を始める前に必ず実行

### ステージ → コミット → Push

自分の変更をチームに共有します。

```
1. ファイルを編集・保存（Cmd+S / Ctrl+S）

2. Source Controlパネルを開く
   └─ 変更ファイルが「Changes」に表示される

3. ファイル横の「+」をクリック（ステージ）
   └─「Staged Changes」に移動する

4. メッセージを入力（例：「1/29 議事録追加」）

5. 「✓」をクリック（コミット）

6. 「...」→「Push」または左下の「🔄」
```

### ファイルの状態マーク

| マーク | 意味 |
|--------|------|
| **U** | 新規ファイル（Untracked） |
| **M** | 変更あり（Modified） |
| **D** | 削除された（Deleted） |
| **A** | ステージ済みの新規ファイル（Added） |

## 9. ファイル・フォルダの追加

### 新しいファイルを作成（GitHub Web）

1. 「Add file」→「Create new file」
2. ファイル名を入力（例：`新しいマニュアル.md`）
3. 内容を入力
4. 「Commit changes」をクリック

### フォルダを作成する

ファイル名に `/` を入れるとフォルダが作られます。

```
入力：01_基本マニュアル/はじめに.md

結果：
📁 01_基本マニュアル/
   └─ はじめに.md
```

### ファイルをフォルダに移動する

1. 移動したいファイルを開く
2. 鉛筆アイコン（Edit）をクリック
3. ファイル名欄を編集：
   ```
   変更前：チェックリスト.md
   変更後：04_品質管理/チェックリスト.md
   ```
4. 「Commit changes」

### 簡単な方法：「.」キーでドラッグ＆ドロップ

GitHubのリポジトリページで **キーボードの「.」** を押すと、Web版VSCodeが開きます。
ファイルをドラッグ＆ドロップで移動できます。

## 10. 便利機能

### 「.」キーでWeb版VSCodeを開く

```
GitHubのリポジトリページ
    ↓
キーボードの「.」を押す
    ↓
github.dev（Web版VSCode）が開く
    ↓
ドラッグ＆ドロップでファイル整理
    ↓
Source Controlからコミット
```

### Git Graph（履歴を見る）

拡張機能「Git Graph」をインストールすると、変更履歴が視覚的に見れます。

**インストール方法：**
1. 左サイドバーの拡張機能アイコン
2. 「Git Graph」を検索
3. 「Install」をクリック

### 変更を元に戻す

| 状況 | 操作 |
|------|------|
| 保存前 | `Cmd/Ctrl + Z` |
| 保存後・コミット前 | Source Control → ファイル横の ↩️ |
| コミット後・Push前 | 「...」→「Undo Last Commit」 |
| Push後 | 「...」→「Revert Last Commit」 |

## 11. よくあるトラブルと対処法

### Pushできない

**原因：** 他のメンバーが先に更新している

**対処：**
1. 先にPullを実行
2. 再度Pushする

### 「ステージされている変更がなく、コミットできません」

**原因：** コミットするものがない

**対処：**
1. ファイルを編集して保存
2. Source Controlで「+」をクリック（ステージ）
3. コミットする

### コンフリクト（競合）が発生

**原因：** 同じ行を複数人が同時に編集した

**対処：**
1. VSCodeが該当箇所を表示
2. 「Accept Current」「Accept Incoming」「Accept Both」から選択
3. 保存 → コミット → Push

**予防策：** 作業前に「○○ファイル編集します」と声かけ

### Pushしたのに見れない

**確認すること：**
1. 左下が `0↑` になっているか（`1↑`以上ならまだPush必要）
2. GitHubで正しいリポジトリを見ているか
3. ブラウザをリロードしたか

## 12. チーム運用ルール

### 基本ルール

```
✅ 作業開始前に必ずPull
✅ 編集したらすぐCommit & Push（溜めない）
✅ 同じファイルを同時編集しない（声かけする）
✅ コミットメッセージは何をしたか書く
```

### コミットメッセージの書き方

| 良い例 | 悪い例 |
|--------|--------|
| 「クライアントA 1/29議事録追加」 | 「更新」 |
| 「KW戦略 内部リンク項目追加」 | 「修正」 |
| 「2月進捗レポート作成」 | 「追加」 |

### ファイル命名ルール

```
推奨：
2025-01-29_進捗MTG.md（日付を先頭に）
01_キーワード選定.md（番号で順序付け）

非推奨：
議事録.md（いつのかわからない）
test.md（何のファイルかわからない）
```

## 13. 用語集

| 用語 | 意味 |
|------|------|
| **リポジトリ** | ファイルの保管場所（フォルダのようなもの） |
| **クローン** | リポジトリをPCにダウンロードすること |
| **Pull** | GitHubの変更をPCに反映（ダウンロード） |
| **Push** | PCの変更をGitHubに反映（アップロード） |
| **ステージ** | コミットに含めるファイルを選択すること |
| **コミット** | 変更を記録すること（セーブポイント） |
| **ブランチ** | 作業を分岐させる機能（上級者向け） |
| **コンフリクト** | 同じ箇所を複数人が編集して競合すること |
| **Collaborator** | リポジトリへの編集権限を持つメンバー |
| **マークダウン** | テキストを構造化する記法（.mdファイル） |

## クイックリファレンス

### 毎日の流れ

```
1. Cursor/VSCodeでプロジェクトを開く
2. 🔄 または Pull（最新を取得）
3. ファイルを編集・保存
4. Source Control → + → メッセージ → ✓
5. Push または 🔄
```

### AI活用の流れ

```
1. 案件フォルダを開く
2. AIに指示：「過去の議事録を読んで〇〇して」
3. AIが全ファイルを参照して回答
4. 結果をファイルに保存
5. Commit & Push で共有
```
