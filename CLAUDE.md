# CLAUDE.md — SEOプロジェクト管理リポジトリ

## 業務改善方針（基本原則）

以下の方針は、すべての作業・回答・提案における基本原則である。

### 1. 目的意識の徹底

- すべての作業・提案において「なぜこれを行うのか」という目的を明確にすること。
- 属人性を排除し、無駄な動きや本来不要な工程を提案に含めないこと。
- 施策着手前に以下 4 項目を必ず固定する。これが崩れると下流すべてにブレが生じる。

| 固定項目 | 問い |
|---------|------|
| ターゲット定義と優先度 | 誰に届けるか。優先順位・関心・不安は何か |
| CV 定義と優先度 | ゴールアクションは何か。CV までの心理ステップ |
| 差別化ポイント | 競合との違い。サイト上で何を強調するか |
| 認知経路（流入元） | ユーザーはどこから来るか。経路ごとの着地ページ |

### 2. 数値を根拠とした設計

- 「なんとなく」や感覚的な判断を一切排除すること。
- 商品設計や事業設計において、常に定量的なデータ（数値）を根拠として提示すること。
- 提案・報告・改善には必ず Before → After の定量比較を添える。

| 指標 | 計測元 | 更新頻度 |
|------|--------|---------|
| セッション数 | Google Analytics | 月次 |
| CV 数 | Google Analytics | 月次 |
| CVR（CV ÷ セッション） | GA 算出 | 月次 |
| オーガニック流入 | GA / GSC | 月次 |
| 検索順位・CTR | Google Search Console | 月次 |

- 効果検証には「何が効いたか・想定外の要因・再現性」の定性分析も必ず含める。

### 3. 目的への落とし込みと数値目標の連動

- 目的を達成するためのプロセスには、必ず具体的な数値見込みをセットで検討すること。
- 作業の進捗や成果が、当初の数値目標に対してどう機能しているかを常に意識すること。
- 収支は「粗利 ＝ 売上 − 経費」「粗利率 ＝ 粗利 ÷ 売上 × 100」で追う。
- ライター消化率は「検収済み記事数 ÷ 発注記事数 × 100」で管理する。

### 運用ルール

- ユーザーから曖昧な指示や数値根拠が不足している提案があった場合、上記原則に立ち返り指摘すること。
- 提案時には「目的 → 数値根拠 → 具体施策 → 数値目標」の流れを意識すること。

## このリポジトリの目的

SEOチームの **記事進捗（ライター稼働／納品／検収）** と **案件ごとの売上・経費** を
Markdown + Git で一元管理し、AI が過去の蓄積を参照して業務を加速できる基盤にする。

## 月次クロージング

| タイミング | やること | 更新先 |
|-----------|---------|--------|
| 月初 | 当月の納品予定を入力 | ライター稼働 月別 |
| 随時 | 納品・検収ステータス更新 | 同上 |
| 月末 ① | 案件ごとの原価を確定 | ライター稼働ファイル |
| 月末 ② | 案件売上・経費を転記 | 収支管理/YYYY/MM_収支.md |
| 月末 ③ | 全社ライター消化率を集計 | ライター管理/月別消化率.md |

## ファイル命名規則

- Markdown: `YYYY-MM-DD_内容.md`（日付昇順ソート）
- クライアントフォルダ: `DM{6桁連番}_{クライアント名}`

## Git 運用ルール

- 作業前に必ず Pull。Commit & Push はすぐ行う（溜めない）
- 同じファイルを同時編集しない（Slack で調整）
- コミットメッセージは「何をしたか」がわかるように書く
  - 良い例: `クライアントA 1/29議事録追加`
  - 悪い例: `更新`

## AI 活用の想定ユースケース

- **MTG 準備**: 過去の議事録を分析し、主要懸念と決定事項をまとめる
- **提案作成**: 効果検証データをもとに、次月の提案方向を示す
- **新メンバーオンボーディング**: プロジェクト概要・現状・ネクストアクションを説明する
- **進捗報告**: 効果検証をクライアント報告用にまとめる
- **収支確認**: 案件別の粗利・粗利率を集計し、収益改善ポイントを示す
