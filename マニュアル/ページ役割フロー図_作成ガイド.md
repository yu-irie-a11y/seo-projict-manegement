## 1. Phase 1 — 目的・前提の固定（最重要フェーズ）

このフェーズの目的は、以後の全成果物が依拠する「判断基準」を確定させること。ここで決めた内容は、UX設計ブリーフ・ページ構成図・フロー図のすべてに波及する。

### 1-1. 固定すべき4項目

| # | 固定項目 | 具体的に決めること | 確認相手 |
|---|---|---|---|
| 1 | ターゲット定義と優先度 | 誰がメインターゲットか、優先順位、各ターゲットの関心事と不安 | クライアント |
| 2 | コンバージョン(CV)定義と優先度 | 何をゴール行動とするか、CVの優先順位、CVまでの心理ステップ | クライアント |
| 3 | 差別化ポイント | 競合との違い、サイト上で強調すべきこと | クライアント |
| 4 | 認知経路（流入元） | ユーザーはどこからサイトに来るか、各経路の着地ページ | クライアント＋データ |

### 1-2. 実施手順

#### ステップ1: ヒアリング議事録から情報を抽出

過去の議事録（`02_議事録/`）から以下を拾い出す:

- クライアントが語った「うちの強み」
- 「どんな人に来てほしいか」の発言
- 「最終的にどうしてほしいか」の発言
- 競合として意識している他社/他校名

#### ステップ2: ターゲットを優先度順に定義

**出力形式:**

| 優先度 | ターゲット | サイトでの主な行動 |
|:---:|---|---|
| 1位 | （例: 中学受験を検討する小5-6保護者） | コース比較 → 説明会予約 |
| 2位 | ... | ... |
| 3位 | ... | ... |

加えて、各ターゲットごとに「関心事（知りたいこと）」と「不安・障壁」を表で整理する。

> **判断基準**: 1位・2位のターゲットに最適化する。3位以降は「迷わず辿り着ける」を最低条件とし、動線の主役にはしない。

#### ステップ3: CV行動を優先度順に定義

**出力形式:**

| 優先度 | CV行動 | 理由（なぜこのCVが重要か） |
|:---:|---|---|
| 1位 | （例: 説明会予約） | 来校が最も入学決定に直結する |
| 2位 | ... | ... |

#### ステップ4: 差別化ポイントを整理

競合名を挙げたうえで、自社/自校の優位点を列挙する。

**出力形式:**

| 差別化ポイント | 訴求するコンテンツ |
|---|---|
| （例: 少人数制・面倒見の良さ） | 学びの特色ページ、在校生の声 |
| （例: 放課後学習環境（自習室20時まで）） | コース詳細ページ、FAQ |

#### ステップ5: 認知経路を一覧化

**出力形式:**

| 経路 | 想定着地ページ | 重要度 |
|---|---|---|
| （例: 塾からの紹介） | トップページ or コース詳細 | 高 |
| （例: Google検索「〇〇 学費」） | 学費ページ直接 | 高 |

### 1-3. 参考実例: 金光八尾の場合

以下は実際のプロジェクトで固定した内容（`金光八尾_UX設計ブリーフ_2026-02-07.md` より抜粋）。

**ターゲット優先度:**

| 優先度 | ターゲット | サイトでの主な行動 |
|:---:|---|---|
| 1位 | 中学受験を検討する小5-6保護者 | コース比較 → 説明会予約 / 資料請求 |
| 2位 | 高校受験を検討する中3生とその保護者 | コース比較 → 説明会予約 / 資料請求 |
| 3位 | 在校生の保護者 | お知らせ確認 / 行事予定 / 書類DL |

**CV優先度:**

| 優先度 | CV | 理由 |
|:---:|---|---|
| 1位 | 説明会予約 | 来校が最も入学決定に直結する |
| 2位 | 資料請求 | 説明会に来られない層の受け皿 |
| 3位 | イベント予約 | 説明会より気軽な初接触 |
| 4位 | お問い合わせ | 検討度は高いがCV率は低い |

**CVまでの心理ステップ（ファネル）:**

```
知る（認知）       気になる（興味）     比べる（比較）       安心する（解消）     申し込む（行動）
「どんな学校？」   「コースが面白そう」  「他校とどう違う？」  「学費は？いじめは？」 「説明会に行こう」

トップページ       コース詳細           合格実績             FAQ                  説明会予約
学校紹介TOP        学びの特色           合格体験記           学費                 資料請求
                   在校生の声           他校との差別化        在校生の声           お問い合わせ
```

### 1-4. Phase 1 完了ゲート

以下の全項目に ✅ が入るまで Phase 2 に進んではならない。

- [ ] ターゲットが優先度順に定義されている
- [ ] 各ターゲットの「関心事」と「不安・障壁」が列挙されている
- [ ] CV行動が優先度順に定義されている
- [ ] CVまでのファネル（心理ステップ）が言語化されている
- [ ] 競合との差別化ポイントが明文化されている
- [ ] 認知経路（流入元）と想定着地ページが整理されている
- [ ] 上記がクライアントと合意されている（議事録 or 承認記録あり）

> ⚠️ このゲートを通過せずにPhase 2以降に進むことは禁止。
> 目的の前提が崩れると、その後の全工程で大きなブレが生じる。

## 2. Phase 2 — UX設計ブリーフの作成

Phase 1で固定した目的を「設計判断の根拠」として文書化する。デザイナー・コーダー・クライアントの三者が参照する共通言語となる。

参考: `金光八尾_UX設計ブリーフ_2026-02-07.md`

### 2-1. ブリーフに含める6セクション

| # | セクション | 内容 | Phase 1との関係 |
|---|---|---|---|
| 1 | ユーザー定義 | ターゲット優先度、関心事・不安、認知経路 | 項目1,4をそのまま展開 |
| 2 | コンバージョン定義 | CV優先度、ファネル5段階 | 項目2を展開 |
| 3 | ユーザーフロー | 推奨動線（最低3本、推奨5本） | 新規作成 |
| 4 | ページ役割一覧 | 全ページのファネル位置・流入元・ゴール | 新規作成 |
| 5 | お客様確認チェックリスト | 認識合わせの確認表 | Phase 1合意の補強 |
| 6 | 次のアクション | 担当・期限 | 進行管理 |

### 2-2. ファネル5段階の標準定義

全成果物で共通して使うファネル表記。案件ごとにカスタマイズ可能だが、5段階構造は維持する。

| 段階 | 意味 | 保護者の心理（学校サイトの場合） | 対応ページ例 |
|---|---|---|---|
| 知る | 初めてサイトに触れる | 「どんな学校？」 | TOP、学校紹介 |
| 気になる | 興味を持ち深掘りする | 「コースが面白そう」 | コース詳細、学校生活 |
| 比べる | 他と比較検討する | 「他校とどう違う？」 | 合格実績、体験記 |
| 安心する | 不安を解消する | 「学費は？雰囲気は？」 | FAQ、学費、在校生の声 |
| 申し込む | 行動を起こす | 「説明会に行ってみよう」 | 説明会予約、資料請求 |

### 2-3. ユーザーフローの書き方

各フローは以下の形式で記述する。

```
[流入元]
  → [着地ページ]
    → [中間ページ1]
      → [中間ページ2]
        → [CV] ★ゴール

設計ポイント:
- （このフローで特に重要な判断）
- （配置すべきCTAの指針）
```

**必須フロー:**
1. メインターゲットの王道動線（最重要）
2. サブターゲットの王道動線
3. 不安解消経由の動線

**推奨フロー（あれば追加）:**
4. イベント駆動型（最短動線）
5. リピーター動線（再訪者向け）

### 2-4. ページ役割一覧の書き方

全ページをテンプレートタイプ別にグルーピングし、以下の表で整理する。

| P# | ページ名 | ファネル | 主な流入元 | ページゴール（次の1アクション） |
|:---:|---|:---:|---|---|

**ルール:**
- 全ページに「次の1アクション」を1つ定義する（複数ある場合は最も重要なものを太字にする）
- CVに直接関係しないページ（在校生向け、法的要件等）は「対象外」と明記する
- セクションハブ型のページは「振り分け」をゴールとして記載する

### 2-5. Phase 2 完了ゲート

- [ ] ファネル5段階に全ページが分類されている
- [ ] 推奨ユーザーフローが3本以上定義されている
- [ ] 全ページに「次の1アクション（ページゴール）」が定義されている
- [ ] お客様確認チェックリストが準備されている
- [ ] サイトマップとの差分分析が完了している（新規/削除ページの特定）
- [ ] ブリーフがクライアントに共有可能な状態になっている

## 3. Phase 3 — テンプレート分類・ページ構成図・リンク設計書

UX設計ブリーフを「具体的なページの中身」に落とし込む。3つの成果物を作成する。

### 3-1. 3つの成果物

| # | 成果物 | 目的 | 参考ファイル |
|---|---|---|---|
| 1 | ウェブサイトテンプレート分類 | 全ページを10〜15のテンプレートに分類 | `金光八尾_ウェブサイトテンプレート分類_2026-02-07.md` |
| 2 | ページ構成図 | 全ページのセクション構成と要素を図示 | `金光八尾_ページ構成図_2026-02-07.md` |
| 3 | リンク設計書 | 全ページのリンク要素と遷移先を一覧化 | `金光八尾_リンク設計書_2026-02-07.md` |

### 3-2. テンプレート分類の手順

1. 現行サイトの全ページをレイアウトパターン（カラム数、ヒーロー有無、サイドバー有無）で分類
2. 各テンプレートに「特徴」「該当ページ一覧」「推定ページ数」を記録
3. 共通要素（ヘッダー、フッター、追従CTA）は全テンプレート共通として別途記載

金光八尾の場合は11テンプレートに分類（トップページ型、セクションハブ型、詳細コンテンツ型、フォーム型など）。

### 3-3. ページ構成図の記法

各ページを縦スクロール形式で記述。セクションごとに以下の要素を記載する:

- `[画像]` — 画像・ビジュアル要素
- `【 】` — CTAボタン（行動ボタン）。赤字で表記
- `→` — テキストリンクの遷移先
- `(外部)` — 外部サイトへのリンク
- `[PDF]` — ダウンロード要素

### 3-4. リンク設計書の記法

ページごとにテーブル形式で記載:

| # | リンク要素 | 遷移先 | 種別 |
|---|---|---|---|
| 1 | 「説明会予約」ボタン | 説明会予約フォーム | CTA |
| 2 | コース名テキスト | /junior/course/s-tokushin/ | 内部 |

共通リンク（グローバルナビ、フッター）は先頭で一括記載し、各ページでは省略する。

- [ ] 全ページがテンプレートタイプに分類されている
- [ ] ページ構成図が全ページ分作成されている
- [ ] リンク設計書が全ページ分作成されている
- [ ] 共通要素（グローバルナビ、フッター、追従CTA）が文書化されている
- [ ] ページ構成図とリンク設計書が相互参照されている
- [ ] CVに関わる行動ボタンが全ページで特定されている

## 4. Phase 4 — ページ役割フロー図の制作

Phase 2（UX設計ブリーフ）と Phase 3（ページ構成図・リンク設計書）を統合し、インタラクティブなHTML成果物を作成する。

参考: `金光八尾_ページ役割フロー_2026-02-07.html`

### 4-1. フロー図の概要

- サイドバーでページを選択すると、そのページからCV（説明会予約等）までの推奨ルートがタイムライン形式で表示される
- 単体HTMLファイル（外部依存なし）としてブラウザで閲覧可能
- お客さま向け / 制作向けの2モード切り替え

### 4-2. データ構造

各ページは以下のJavaScriptオブジェクトで定義する:

```javascript
{
  id: Number,       // ページ番号（P01なら1）
  n: String,        // ページ名
  u: String,        // URL
  t: String,        // テンプレートタイプ
  f: String,        // ファネル位置（知る/気になる/比べる/安心する/申し込む）
  r: String,        // 役割説明文
  s: Array,          // セクション配列
  // s: [{n:"セクション名", d:["要素1","要素2"]}]
  c: Array,          // 行動ボタン含有セクションのindex
  ei: Number,        // 出口セクション（次ページへの遷移元）のindex
  ni: Number,        // 次ページのID（単一遷移の場合）
  na: String,        // 次アクションの説明文
  // --- 分岐がある場合 ---
  br: Array,         // 分岐配列
  // br: [{l:"分岐ラベル", sub:"心理説明", ni:nextId, na:"アクション説明"}]
  // --- CVページの場合 ---
  cv: Boolean,       // true = CVページ
  cvName: String,    // CV名（「説明会予約」等）
  // --- オフフローページの場合 ---
  off: String        // 表示ラベル（「在校生向け」等）
}
```

### 4-3. 制作手順

#### Step 1: プロトタイプ作成

サンプル4〜5ページでUI・データ構造を確認する。含めるべきページ:
- 入口ページ（例: 中学校TOP）
- 中間ページ（例: 学校生活）
- 分岐があるページ（例: 学校生活→クラブ or 直接CV）
- CVページ（例: 説明会予約）

#### Step 2: 全ページデータ投入

UX設計ブリーフのページ役割一覧を基に、全ページのデータを投入する。入力元の対応:

| フィールド | 入力元 |
|---|---|
| `t`（テンプレート） | テンプレート分類 |
| `f`（ファネル） | UX設計ブリーフ Section 4 |
| `r`（役割） | UX設計ブリーフ Section 4「ページゴール」 |
| `s`（セクション） | ページ構成図 |
| `ni`/`br`（遷移先） | リンク設計書 + ユーザーフロー |

#### Step 3: フローチェーン検証

全ページからCVまでの経路が途切れていないことを検証する。

```javascript
// 検証スクリプトの考え方
// 各ページを起点にフローを辿り、CVページに到達するかを確認
// 分岐がある場合は全分岐パターンを検証
// オフフローページ（在校生向け等）は検証対象外
```

**合格基準:**
- 全フローページからCVに到達する
- 全next-id参照が有効（存在しないIDを指していない）
- 全72ページにセクションデータがある

### 4-4. デュアルモード設計

| モード | 対象 | 表示内容 | 非表示 |
|---|---|---|---|
| お客さま向け | クライアント | 役割、ファネル、行動ボタン、遷移先名 | URL、画像/カード詳細、ATF境界線 |
| 制作向け | デザイナー/コーダー | 上記すべて + URL、全要素、ATF境界線、追従CTA仕様 | — |

### 4-5. Phase 4 完了ゲート

- [ ] 全ページのデータが入力されている
- [ ] サイドバーに全ページが表示されている
- [ ] 各ページからCVまでのフロータイムラインが表示される
- [ ] 分岐がある全ページでタブ切り替えが機能する
- [ ] お客さま向け / 制作向けの切り替えが機能する
- [ ] オフフローページが明示的にラベルされている
- [ ] ファネル色が正しく表示されている（5色: 青/緑/黄/紫/赤）
- [ ] ブラウザで単体HTMLファイルとして開ける（外部依存なし）
- [ ] フローチェーンに途切れがない（全ルートがCVに到達）

## 5. Phase 5 — レビュー・フィードバック・修正

### 5-1. レビュー体制

| レビュー | 確認者 | 確認観点 |
|---|---|---|
| 社内レビュー | ディレクター | ファネル整合性、動線の抜け漏れ |
| デザイナーレビュー | デザイナー | セクション構成の実現性、制作向け情報の過不足 |
| クライアントレビュー | 担当者 | 目的との整合性、役割定義の妥当性 |

### 5-2. レビュー手順

1. GitHub PagesのURLを共有し、ブラウザで操作しながら確認してもらう
2. フィードバックを議事録に記録
3. HTMLファイルを修正し、GitHub Pagesに再デプロイ（Phase 6手順参照）
4. 必要に応じてUX設計ブリーフにもフィードバックを反映

### 5-3. 修正時の巻き戻しルール

| 修正の影響範囲 | 対応 |
|---|---|
| 表示・UIの調整のみ | Phase 4内で対応 |
| セクション構成の変更 | Phase 3のページ構成図も更新 |
| ユーザーフロー・CV定義の変更 | **Phase 1に戻る**。ゲートを再通過する |
| ターゲット定義の変更 | **Phase 1に戻る**。全成果物を見直す |

> Phase 1に影響する修正が入った場合、その後の全成果物を確認・更新する必要がある。
> これが「目的を最初に固定する」ことの意味である。

### 5-4. Phase 5 完了ゲート

- [ ] 社内レビュー完了
- [ ] デザイナーレビュー完了（制作向け表示の確認）
- [ ] クライアントレビュー完了
- [ ] フィードバック全件対応済み
- [ ] 修正後のフローチェーン再検証完了

## 6. Phase 6 — GitHub Pagesでの共有・納品

### 6-1. 公開用リポジトリの作成手順


## 9. トラブルシューティング

| 問題 | 原因 | 対処 |
|---|---|---|
| フローチェーンが途切れる | 中間ページの `ni` / `br` が未設定 | データの ni / br フィールドを確認・修正 |
| ファネル色が表示されない | `f` フィールドの値が5段階と不一致 | 知る/気になる/比べる/安心する/申し込む のいずれかに統一 |
| 分岐タブが表示されない | `br` 配列が未定義 | br フィールドに分岐データを追加 |
| GitHub Pagesに反映されない | push していない or キャッシュ | push 確認後、ブラウザでハードリロード（Ctrl+Shift+R） |
| HTMLファイルが他人のPCで開けない | 日本語ファイル名の文字化け | `index.html` に英数字リネームして zip 共有、または GitHub Pages を使用 |
| セクション構成が見づらい | 役割とセクションの境界が不明瞭 | 「ページ構成」枠（sections-frame）でセクション群を囲む |
